# OpenCode Mermaid Preview Plugin Design

Date: 2026-02-27
Status: Approved
Scope: v1

## Goal

Build a simple OpenCode plugin that visualizes Mermaid code generated by models.

Environment behavior:
- TUI: auto-detect Mermaid in model responses and preview in-terminal.
- Web/Desktop: show a toggle at the bottom of the chat message to preview Mermaid inline.

Out of scope for v1:
- Mermaid from user messages.
- Mermaid from pasted files.

## Architecture

Use a unified core with surface adapters.

Core modules:
- `MermaidBlockDetector`: parse model response text parts and extract fenced `mermaid` blocks.
- `RenderCoordinator`: call `beautiful-mermaid` to render SVG, normalize success/failure, and cache output.
- `PreviewStateStore`: track per-block UI state and render lifecycle.

Surface adapters:
- `TuiAdapter`: auto-detect and auto-preview in terminal when image rendering is supported.
- `ChatAdapter`: add a message-bottom toggle and render inline on demand.

This keeps behavior consistent across surfaces while minimizing duplicated logic.

## Components And Data Flow

1. `ModelMessageWatcher` listens to model response updates only.
2. It sends message text to `MermaidBlockDetector`.
3. Detector emits normalized blocks:
   - `messageId`
   - `partId`
   - `blockIndex`
   - `code`
   - `blockHash`
4. `RenderCoordinator` checks cache by `blockHash`.
5. On cache miss, render Mermaid via `beautiful-mermaid` to SVG.
6. Adapters consume the same result:
   - TUI auto-renders the SVG in terminal image view.
   - Chat shows a toggle; first open triggers lazy render, then cached reuse.
7. Chat path displays SVG via image-safe path (`image/svg+xml` attachment-like rendering), not raw inline `<svg>` markup.

## Error Handling

- Parse issues are non-fatal: leave original text unchanged.
- Render failures show:
  - raw Mermaid block
  - warning message (`Mermaid preview unavailable`)
- TUI capability failures (no image support) use the same fallback.
- One broken block does not break other blocks.
- Streaming updates are hash/debounce aware to reduce flicker and repeated work.
- Optional debug logging includes `messageId`, `blockIndex`, and error type.

## Testing Strategy

Unit tests:
- Detector: single/multiple blocks, mixed fences, incomplete fences, stream updates.
- Coordinator: cache hit/miss, successful SVG render, render failure fallback.

Adapter tests:
- TUI: auto-detect, supported terminal render, unsupported terminal fallback.
- Chat: toggle visibility, lazy first render, reopen from cache.

Integration tests:
- End-to-end from model response to preview/fallback for multiple blocks.
- Isolation check: one invalid block does not affect valid blocks.

Acceptance checks:
- Web/Desktop: toggle appears only when Mermaid exists.
- TUI: preview appears automatically when Mermaid exists.
- Failure path always preserves raw block with warning.

## Decisions Captured

- Chosen approach: unified core + adapters.
- Source scope: model responses only for v1.
- TUI trigger: auto-detect.
- Chat trigger: bottom-of-message toggle.
- Render backend: `beautiful-mermaid` for both surfaces.
- Preferred TUI output: SVG in terminal image rendering path.
- Failure behavior: show raw block plus warning.

## Risks And Mitigations

- Terminal image support variance across environments.
  - Mitigation: capability check + raw fallback.
- SVG handling differences in chat rendering path.
  - Mitigation: use image MIME route (`image/svg+xml`) instead of raw inline SVG.
- Streamed partial Mermaid blocks causing noisy rerenders.
  - Mitigation: block-hash based debounce and cache.

## Success Criteria

- Mermaid in model responses is previewable in both TUI and web/desktop according to the chosen UX.
- Fallback behavior is deterministic and non-destructive.
- Plugin remains small, with shared rendering logic and minimal adapter code.
